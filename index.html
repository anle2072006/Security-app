💉 Cross-Site Scripting (XSS)
Kẻ tấn công inject JavaScript độc hại vào trang web. Script chạy trong browser của nạn nhân — đánh cắp cookie, session token, keylog, redirect trang.
⚡ Attack Flow
Attacker gửi input chứa <script> hoặc event handler → Server lưu vào DB hoặc reflect về → Browser parse HTML → Script thực thi → Game over

✅ Hardened
// ✅ Escape HTML entities
function escapeHtml(s) {
  return s
   .replace(/&/g, '&amp;')
   .replace(/</g,  '&lt;')
   .replace(/>/g,  '&gt;')
   .replace(/"/g,   '&quot;')
   .replace(/'/g,   '&#x27;');
}

app.get('/search', (req, res) => {
  const q = escapeHtml(req.query.q);
  res.send(`<p>Kết quả: ${q}</p>`);
});

✓ Escape output HTML (dùng template engine auto-escape: React JSX, Jinja2)
✓ Validate & whitelist input
✓ Content Security Policy (CSP) — block inline scripts
✓ Cookie: HttpOnly + Secure + SameSite=Strict
✓ Dùng textContent thay vì innerHTML trong JS
🗄 SQL Injection
Kẻ tấn công chèn SQL code vào input để thao túng database query. Hậu quả: bypass authentication, dump toàn bộ dữ liệu, xóa bảng, thậm chí RCE.
❌ Vulnerable — String concat
// ❌ Nối string thẳng vào query
const query =
  "SELECT * FROM users " +
  "WHERE username = '" + username + "' " +
  "AND password = '" + password + "'";
Username (thử: admin' -- hoặc ' OR '1'='1): Password:
Query sẽ hiển thị ở đây...
✅ Hardened — Parameterized
// ✅ Prepared statement (mysql2)
const sql =
  "SELECT * FROM users " +
  "WHERE username = ? " +
  "AND password = ?";

db.execute(sql, [username, password]);

// ✅ Hoặc dùng ORM (Prisma)
await prisma.user.findFirst({
  where: { username, password }
});
Same payload — parameterized:
Query sẽ hiển thị ở đây...

🎭 CSRF — Cross-Site Request Forgery
Kẻ tấn công lừa browser của người dùng đã đăng nhập gửi request tới server mà không có sự đồng ý. Browser tự động đính kèm cookie → server tin là request hợp lệ.

Attack Flow:

User login bank.com
→
Visit evil.com
→
Hidden form auto-submit
→
POST bank.com/transfer
→
Money transferred! 💸
❌ evil.com/attack.html
<body onload="document.forms[0].submit()">

<form action="https://bank.com/transfer"
      method="POST">
  <input type="hidden"
    name="to_account"
    value="attacker_acc"/>
  <input type="hidden"
    name="amount"
    value="50000000"/>
</form>
⚡ Tại sao thành công?
Browser tự gửi session cookie của bank.com khi POST → Server không phân biệt được request từ bank.com hay evil.com
✅ CSRF Token Defense
// Server dùng csurf middleware
const csrf = require('csurf');
app.use(csrf({ cookie: true }));

// Template nhúng token ẩn
<form action="/transfer" method="POST">
  <input type="hidden"
    name="_csrf"
    value="{{ csrfToken() }}"/>
</form>

// Cookie config
res.cookie('session', token, {
  httpOnly: true,
  secure: true,
  sameSite: 'Strict'
});
✅ Tại sao an toàn?
evil.com không biết CSRF token (random, per-session) → không thể forge request. SameSite=Strict block cookie cross-site.
🛡 Defense Strategies
1. CSRF Tokens — random token per session, validate mỗi state-changing request
2. SameSite Cookie — SameSite=Strict/Lax ngăn cookie gửi cross-site
3. Double Submit Cookie — gửi token cả trong cookie và request body
4. Custom header — AJAX: X-Requested-With: XMLHttpRequest
📋 Content Security Policy (CSP)
HTTP response header khai báo policy tài nguyên cho browser. Là lớp bảo vệ thứ 2 — nếu XSS xảy ra, CSP ngăn script độc hại chạy và connect ra ngoài.
📄 Ví dụ CSP Header (Express + Helmet)
const helmet = require('helmet');

app.use(helmet.contentSecurityPolicy({
  directives: {
    defaultSrc:  ["'self'"],
    scriptSrc:   ["'self'", "https://cdn.jsdelivr.net"],
    styleSrc:    ["'self'", "https://fonts.googleapis.com"],
    imgSrc:      ["'self'", "data:", "https:"],
    connectSrc:  ["'self'", "https://api.myapp.com"],
    objectSrc:   ["'none'"],
    frameAncestors: ["'none'"],
    upgradeInsecureRequests: [],
    reportUri:   ["/csp-report"],
  },
}));

Các directive quan trọng:

default-src'self'Fallback cho mọi resource type chưa khai báo
script-src'self' cdn.js.netKiểm soát nguồn script. Tránh 'unsafe-inline'
style-src'self' fonts.google.comWhitelist CSS sources
img-src'self' data: https:Cho phép image từ HTTPS và data URI
connect-src'self' api.myapp.comKiểm soát fetch, XHR, WebSocket
object-src'none'Block Flash/plugins — luôn set 'none'
frame-ancestors'none'Ngăn Clickjacking (thay thế X-Frame-Options)
report-uri/csp-violationsBrowser gửi report khi vi phạm CSP
❌ Không có CSP
Nếu XSS xảy ra, script có thể load từ bất kỳ domain, gọi bất kỳ API, đánh cắp toàn bộ dữ liệu
✅ Có CSP + XSS xảy ra
Browser block script không có nonce/hash, block kết nối ra ngoài whitelist → thiệt hại bị giới hạn tối đa

🔒 Security Headers
Các HTTP response header bổ sung giúp browser bật thêm lớp bảo vệ. Cấu hình bằng helmet (Node.js) hoặc trong web server (Nginx/Apache).
✅ Helmet.js — bật tất cả 1 dòng
const helmet = require('helmet');
app.use(helmet()); // Bật tất cả security headers mặc định
HeaderGiá trị khuyên dùngTác dụng
Strict-Transport-Security max-age=31536000; includeSubDomains Bắt buộc HTTPS, chống SSL stripping
X-Content-Type-Options nosniff Ngăn MIME sniffing
X-Frame-Options DENY Chống Clickjacking
Referrer-Policy strict-origin-when-cross-origin Kiểm soát thông tin Referer
Permissions-Policy camera=(), microphone=(), geolocation=() Disable browser API nhạy cảm
Cache-Control no-store Ngăn browser cache dữ liệu nhạy cảm
⚠ Lưu ý quan trọng
Security headers là defense in depth — lớp bổ sung, không thay thế việc fix lỗ hổng ở code. Luôn ưu tiên fix root cause (escape output, parameterized query) trước.
